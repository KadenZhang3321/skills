# 问题分类判断树

当面对 CI 失败时，按以下顺序判断问题类型。

## 第一步：是否是基础设施问题？

**看这些信号（任一满足 → 倾向于类型 2）：**

| 信号 | 判断依据 |
|------|---------|
| 失败发生在"环境准备"阶段 | 如 `Preparing environment`、`Downloading dependencies`、`Setting up runner` |
| 网络/存储相关报错 | `connection timed out`、`no space left on device`、`artifact upload failed`、DNS 解析失败 |
| 多个 PR 同时失败 | 说明不是代码问题，而是共享资源问题 |
| 重跑后通过 | 强烈信号：基础设施 flaky，不是代码 bug |
| 等待时间异常长 | CI job 长时间处于 pending/queued 状态，最终 timeout |

**反向排除：**
- 如果错误发生在"Running tests"或"Build"阶段，且有具体的代码行报错 → 不是基础设施问题

---

## 第二步：代码问题 - 已知还是未知？

**已知问题信号（类型 1.1）：**
- 错误信息中出现了你在 issue tracker 里能找到的关键词
- 失败的测试在过去 30 天内曾多次失败
- 该文件/模块有标注 `TODO`、`FIXME` 或相关 open issue
- 错误模式与某个 open issue 的描述高度匹配

**未知问题信号（类型 1.2）：**
- 失败的测试或错误路径直接指向 PR 新增/修改的代码
- CI 在 PR 创建之前是绿的（如果能确认）
- 其他人的 PR 没有同样的失败

---

## 第三步：还是搞不清楚 → 类型 3

同时存在多个方向的信号，或者信息不足以判断，归入类型 3，明确列出两个方向各自的证据和待确认项。

---

## 常见混淆场景

**场景 A：测试代码本身有 bug，还是被测代码有 bug？**
- 先看失败的断言是什么
- 如果断言的预期值看起来"不合理"，可能是测试写错了（类型 1.2，修测试）
- 如果断言合理但实际值错了，是被测代码的 bug（类型 1.2，修代码）

**场景 B：依赖下载失败 - 是网络问题还是版本锁定问题？**
- 如果报 "version not found" 或 "package does not exist"，可能是版本号写错（类型 1.2）
- 如果报 "connection refused" 或超时，是网络/基础设施问题（类型 2）

**场景 C：lint 失败**
- 如果 lint 规则是项目的，且代码确实违反了规则 → 类型 1.2（PR 的代码问题）
- 如果 lint 工具本身版本变了导致新规则 → 可能是类型 1.1（已知的升级问题）或类型 3

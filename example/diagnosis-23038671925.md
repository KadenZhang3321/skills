## 根因分析

**故障定性：基础设施 / K8s 容器运行时异常**
**置信度：高**
**责任方：基础设施团队 / CI 维护者**

### 报错原文

```
Executing the custom container implementation failed.
Please contact your self hosted runner administrator.

Error: failed to run script step: Error: command terminated with non-zero exit code:
command terminated with exit code 255
```

### 判定依据

1. **全部上游步骤成功**
   所有 Job 均通过了 Set up job → Initialize containers → Checkout → Install dependencies 等环境准备步骤，说明容器本身可以正常拉起和初始化。

2. **跨版本同步失败**
   `4034c3d`（vllm main）和 `v0.16.0` 两个完全不同的 vllm 版本，在完全相同的步骤以完全相同的方式失败，代码差异无法解释此现象。

3. **跨 Job 类型一致**
   singlecard、multicard-2、multicard-4 三种硬件配置全部失败。

4. **exit code 255**
   容器在测试执行阶段被强制终止（非正常退出），典型原因：NPU 驱动崩溃、节点级 OOM Killer、或 ARC 容器编排脚本与 K8s API Server 断连。

### 结论

此次失败与 `refactor_moe` 分支的代码变更**无关**，是测试执行阶段 Runner 节点发生了底层环境故障。

## 修复建议

### 立即操作

1. **重触发 CI** — 验证是否为瞬时故障
2. **获取 Runner 名，定位物理节点**
   ```bash
   gh api repos/vllm-project/vllm-ascend/actions/jobs/66912284656 \
     --jq '.runner_name'
   ```
   拿到 Runner Pod 名后：
   ```bash
   kubectl get pod <runner_name> -o custom-columns=NODE:.spec.nodeName \
     --kubeconfig=<集群kubeconfig>
   ```
3. **检查 NPU 健康状态**
   确认该时段节点的 `npu-smi info` 记录，查看是否有 `ERR99999` 或驱动报错

### 预防建议

- 为测试容器配置合理的超时和 OOM 告警
- 在 Loki/ELK 中配置 exit code 255 的自动告警规则
